## Glossary

Terms used across the codebase. Definitions are domain-meaningful, not implementation-detail. If a term doesn't carry weight beyond "the obvious" ‚Äî don't add it.

### Core entities

- **Pillar** ‚Äî a multi-milestone capability area. Lives forever in memory, never closes. Narrative grouping only; not a structural unit. Examples: Memory, Autonomy, Identity, Multi-agent. **A pillar is not a task** ‚Äî closing one milestone within a pillar doesn't close the pillar.
- **Milestone** ‚Äî GitHub milestone grouping ‚â•2 capability-coherent slices that share a goal or have inter-dependencies. Description carries the PRD or PRD-equivalent (output of `/grill-me` + `/to-prd`). Closes when capability ships, not on a date ‚Äî no date in title; 0 open issues + state=open is a bug. Term "epic" is **not** used; milestone is the single grouping primitive (`milestone_hierarchy_v3`).
- **Slice** ‚Äî one PR, vertical (schema ‚Üí service ‚Üí API ‚Üí UI ‚Üí tests). A single independent slice with no inter-deps ships **without** a milestone ‚Äî no ceremony for one-offs.
- **Skill** ‚Äî atomic, reusable agent capability defined in `.claude-userlevel/skills/<name>/SKILL.md`. Universal (not project-specific). Loaded by Claude Code at session start.
- **Subagent** ‚Äî agent dispatched via the `Agent` tool from a parent session. Runs in isolation (own context, own worktree if requested), reports back. Not the same as "skill".
- **Memory** ‚Äî durable cross-session knowledge in Supabase. Types: `user`, `project`, `decision`, `feedback`, `reference`. Always carries `source_provenance`.
- **Recall** ‚Äî the pipeline that turns a query into ranked Memory hits: rewriter ‚Üí embed ‚Üí semantic + keyword search ‚Üí RRF merge ‚Üí temporal scoring ‚Üí link expansion ‚Üí known-unknown gate. Lives in `mcp-memory/recall.py` as the deep module behind every recall call site (MCP `recall` tool, PreToolUse hook, eval harness). Three adapters, one implementation.
- **RecallConfig** ‚Äî frozen dataclass of pipeline toggles + constants (`use_rewriter`, `use_links`, `use_classifier`, `use_temporal`, thresholds, RRF-K, temporal half-lives, excluded tags). Prod uses `PROD_RECALL_CONFIG` (all-on defaults); eval flips flags for ablation. Adding a recall feature = one new flag in this dataclass, no fanout across hook/server/eval.
- **RecallHit** ‚Äî structured result row: raw memory + `semantic_score`, `keyword_score`, `rrf_score`, `temporal_score`, `final_score`, `source` (`semantic|keyword|linked`), `linked_via`. Formatting (TextContent vs brief markdown vs eval JSON) is per-adapter, not per-pipeline.
- **Outcome** ‚Äî recorded result of a delegated task / decision, used by `/reflect` and `/verify` to attribute success/failure back to reasoning. Linked to `decision_made` episodes.
- **Decision** ‚Äî a `decision_made` episode emitted via `record_decision`. Captures rationale + alternatives + memories used + reversibility. Trigger conditions in CLAUDE.md memory rules.
- **FOK** (First-of-Kind) ‚Äî a memory recall calibration metric. Indicates how often a recall returns a memory the agent has never seen before. Pillar-1 quality signal.
- **Episode** ‚Äî a structured event in the memory layer (decision, recall, outcome). Each has UUID; cross-references via UUID, not name.
- **Goal** ‚Äî a strategic priority registered in the goals table. Drives Jarvis's autonomous decisions about what to do first.
- **Deriver** ‚Äî per-session-end implicit-memory pass (Workshop+Ollama, DeepSeek fallback). Reads scrubbed transcript, emits ‚â§5 candidate memories per run with `requires_review=true`. Owner-level, runs on every session regardless of project; candidates self-classify scope (`user` ‚Üí global, `feedback` ‚Üí session project or global per content). Honcho analog.
- **Dreamer** ‚Äî scheduled cross-corpus consolidation pass. Triggers on pending-candidate count ‚â•30 OR ‚â•7d since last run. Reads pending + accepted `feedback` from last 90d (cap 200), emits new candidates and merge proposals ‚Äî both gated. Owner-level, single pass across all projects.
- **Candidate** ‚Äî memory row with `requires_review=true`, not yet accepted by owner. Hidden from default recall; opt-in via `include_unreviewed=true`. Promoted to live memory by `memory_review_decide(action=accept)`.
- **Merge proposal** ‚Äî Dreamer-emitted candidate with non-empty `merge_targets UUID[]`. Recall MUST skip these even when `include_unreviewed=true` ‚Äî they are meta-rows, not knowledge. Atomic accept via `memory_review_decide(action=merge_into)` writes new memory + sets `superseded_by` on targets.
- **Curation** ‚Äî owner-invoked weekly hygiene pass over memories. Surfaces candidate stale rows (always_load + content-stale heuristics), owner confirms per-row, writes `expired_at` or `superseded_by`. Skill `/curate`, MCP tools `memory_mark_stale` + `memory_unmark_stale`. Distinct from autonomous consolidation (`memory-consolidation-weekly`) which is LLM-driven and never user-confirmed. Host-only via service-role ‚Äî sandcastle anon RLS blocked. Spec: M45 (#768). Decisions `cbaf47ce-9217-40da-923a-b8edce9f233d` + `d5bfd444-78f3-4fca-bcd8-f392f647504c`.
- **Banner blindness** ‚Äî failure mode where automated hygiene/dismissal hints fire too often ‚Üí user dismisses all of them, including rare legitimate ones. Predicted by the 2026-04-15 `no_memory_hygiene_tool` rationale (UUID `719fb533`); preserved as the invariant that gates M45's owner-invoked-only Curation surface (autonomous detector hook explicitly dropped from the plan).
- **Access-bias** ‚Äî feedback loop where session-start auto-load bumps `last_accessed_at`, ACT-R access boost lifts the row's rank in recall, recall surfaces it, bump compounds. Discovered as the mechanical root cause of #641 always_load misuse. Fixed in M45/S2 (#767) by splitting query-driven vs auto-loaded touch sites at gather time. Catalog already excludes itself (`scripts/session-context.py:355-358`); S2 extends the same exclusion to always_load / user_profile / working_state auto-load sections + `handlers/memory.py:236` touch site.
- **Recall axis** ‚Äî eval-set taxonomy in `tests/memory-eval/queries.yaml`: `axis: lexical` queries probe tokenization / embedding asymmetries (slug-form names, identifiers with `_` / `-`); `axis: lifecycle` queries probe staleness / supersession; `axis: hybrid` covers natural-language prompts that may exercise both. Mixing axes under one expansion conflates failure modes ‚Äî queries.yaml separates them. Must_not violations measured at @5 AND @10 (top-5-only lens hides rank-6-10 regressions).
- **Always-gate** ‚Äî review policy: every Deriver/Dreamer write requires explicit owner accept before influencing recall. No auto-promote tier in v1; future tiered policy must be data-driven from accumulated review decisions, not prompt-derived.
- **Sandcastle** ‚Äî Docker-isolated AFK coding-agent runtime (epic #534). One iteration per container: pick a `sandcastle`-labelled issue, work it on local Ollama, **open a PR but never merge** (decision `436f9549`). Sterile image ‚Äî no `~/.claude` mount, all skills + memory MCP baked in (decisions `894ac658`, `228a2d9b`). Worktree is copy-on-write, so runtime overwrites of tracked files (e.g. `.mcp.json`) don't leak to the host.
- **Watchdog** ‚Äî PowerShell wrapper around a sandcastle run. Auto-starts Docker + Ollama with bounded poll, parses iteration result, writes the `outcome_record` row, fires Telegram only when infrastructure cannot come up. Single command interface, large hidden surface ‚Äî qualifies as a deep module.
- **Safe-hours window** ‚Äî clock-bound interval (e.g. 22:00‚Äì08:00) during which AFK loops may run on Workshop PC. Enforced by **soft-stop**: no kill mid-iteration, just refuse to start a new one once the window closes. Loss-of-WIP avoidance, not strict scheduling.
- **Sandcastle model tier (Workshop, 2026-05-13)** ‚Äî production primary = `qwen2.5-coder:14b`, downgrade Tier 1 = `qwen2.5-coder:7b`. RTX 5080 has only 16 GB VRAM, so any 30B+ Q4 model spills to CPU and runs at ~5 tok/s (unusable). 14b stays VRAM-resident at ~94 tok/s warm. AFK viability threshold is **‚â• 30 tok/s sustained**. Full benchmark + reasoning: [`docs/agents/ollama-workshop-bench-538.md`](docs/agents/ollama-workshop-bench-538.md). The watchdog itself stays model-agnostic; defaults are passed by `/setup-tasks` Task Scheduler entries (#545 / #546).
- **Reactive-core model split (2026-05-24)** ‚Äî two consumers on two substrates (resolves spike #737, decision `2a3a6828-b440-4417-a0b2-e23808767a0b`). **Orchestrator** (per-wake dedup / score / route one event vs memory + vision + active-tasks) runs on local `gemma4:e4b` ‚Äî native tool-use, cold-wake ~10s / warm-steady ~2s per #738 bench on Workshop RTX 5080. **AFK worker** (drives Claude Code through multi-step tool-use) runs on cloud Claude via the Claude Code gate ‚Äî local models could not sustain it (qwen3-coder:30b experiment failed real Claude-Code tool-use under load). Retires the prior single-substrate assumption. Distinct from "Sandcastle model tier" above: Sandcastle covers the AFK-coding-agent runtime (one container per issue); this entry covers the Reactive-core event loop's two roles. Reversible.
- **PR-rework** ‚Äî the AFK loop's second-pass tick: an open PR carries a negative reviewer signal, orchestrator dispatches sandcastle with `SANDCASTLE_TARGET_PR=<N>` to apply fixes on the existing branch (no new branch, no new PR). Distinct from the initial-implementation tick (issue ‚Üí branch ‚Üí PR). Bounded by max-attempts and scope-creep guards; failure to converge escalates to principal.
- **review_negative event** ‚Äî single canonical row in `events` (`event_type=review_negative`, `severity=medium`) emitted by `event-dispatch.yml` when any reviewer asks for changes. `payload.reviewer_kind ‚àà {human, claude_bot}` discriminates source. Quarantines the only fragile string-match (Claude code-review bot writes issue-comments, not reviews ‚Äî its verdict is parsed inside the workflow, never in the orchestrator). Orchestrator filter stays as one stable predicate as new reviewers are added (decision: `2c87e895-2d3a-4b0b-a727-18935d81a4cd`).
- **AFK system** ‚Äî set of jobs that can run end-to-end without human presence. All questions resolved by sandcastle agent (memory + subagents) or orchestrator; Telegram escalation is last-resort, not normal-path. **Progress metric: the set of task-types that qualify as AFK grows over time** ‚Äî system success is measured by raising what kinds of work need no HITL, not by the count of nightly iterations.
- **`event_queue`** ‚Äî the durable EVENT substrate (live `events` table extended in place; backlog start-clean at cutover). Each row carries explicit state `pending | claimed | processed | parked` (default `pending`) and a unique `dedup_key` (sha256 of identifying fields). Claiming records `claimed_at` / `claimed_by`. `AFTER INSERT` NOTIFY trigger (`notify_events_insert`) fires a wake signal on the `'events'` channel, following the same pg_notify pattern as `notify_events_canonical` (which serves the separate `events_canonical` OTel telemetry log ‚Äî trace_id/actor/action/cost ‚Äî NOT this perception queue). Interface: `claim_next() ¬∑ mark_processed() ¬∑ park() ¬∑ requeue()`. Substrate verdict ‚Äî extend in place, do not greenfield (decision: `2c5384d0`).
- **`task_queue`** ‚Äî the durable TASK substrate (reshaped table; approval-gated columns dropped). FSM: `pending ‚Üí claimed ‚Üí running ‚Üí done | failed | parked`. Carries `priority int`, `assignee text`, `idempotency_key`. Smoke rows deleted at cutover. Interface: `enqueue(priority, assignee) ¬∑ claim_next() ¬∑ transition()`. A retrying producer cannot double-enqueue ‚Äî idempotency-key collision is a no-op.
- **`dedup_key`** ‚Äî event-side identity hash (sha256 of identifying fields). Collapses repeat arrivals of the same GitHub signal into one row; unique constraint enforces this at the schema layer.
- **`idempotency_key`** ‚Äî task-side identity hash. Allows retrying producers to enqueue safely without double-effects.
- **`wake_driver`** ‚Äî the algorithmic loop driver. `LISTEN` on the events channel; on `NOTIFY` cold-boots the orchestrator for the next single `pending` event, then loops as soon as the previous tick finishes. No cron, no resident agent ‚Äî persistent BEHAVIOR, not a persistent PROCESS. Watchdog re-claims `claimed` rows older than threshold so a dead orchestrator never strands work. Replaces the old polling-watcher daemon model (continuous-loop wake ‚Äî decision: `efa255cc`; substrate verdict: `2c5384d0`).
- **`orchestrator`** ‚Äî the per-event router. Interface: `handle_event(event) ‚Üí Decision`. **Phased (decision `e5131e38`, /grill 2026-05-29):** MVP (#744) is **pure-deterministic ‚Äî no local model**; the model layer (gemma4:e4b, open-world fallback for unenumerated events, free-text triage, `build_context(event)` seam, goal-aware priority) is deferred to the #872 model-layer slice, because every *current* event type has a deterministic route and a dormant model would ship unvalidated. **Three routes** (refines `09dc5a60`/`1fb05508`): `handle_inline` (Tier-0 lookup, wrapped by `safety.gate()` ‚Äî Tier 1 degrades to an `owner` row, Tier 2 blocks), `emit_task` (`task_queue` row, `assignee='sandcastle'`, `priority = f(severity)`), `escalate_to_human` (`task_queue` row, `assignee='owner'`, weekend-aware per `weekday_weekend_scheduling_policy`). **`assignee ‚àà {sandcastle, owner}`** ‚Äî `cloud` is a category error (substrate ‚âÝ assignee); the target above an agent's authority is the human, not a cloud model. Unknown `(event_type, severity)` ‚Üí fail-safe `escalate_to_human`. Consumes the `events` table (not `events_canonical`). Subsumes the old polling-watcher daemon (`scripts/orchestrator/watcher.py`) ‚Äî its `review_negative ‚Üí /rework` is now the deterministic route. Model layer, when it lands (#872): cold-wake ~10s / warm ~2s on Workshop ([[workshop_coldboot_tick_bound_738]]), Ollama `keep_alive` ‚â• inter-event idle to stay warm.
- **`executor`** ‚Äî the spawn primitive salvaged from the retired LangGraph dispatcher. `spawn(task)` runs `claude -p` in sandcastle through `_sanitize_env` (strips `ANTHROPIC_API_KEY` / `ANTHROPIC_AUTH_TOKEN` / `CLAUDE_API_KEY` so an autonomous run never bills the API account instead of the Max subscription) and `_resolve_claude_binary`. Fire-and-forget ‚Äî after spawn, the loop is closed externally (Path A below), not internally. Carries the salvaged Tier 0 / 1 / 2 `safety.py` gate (`action_agent_safety_gate_model_v1`).
- **Loop closure (3 paths)** ‚Äî the orchestrator owns triage and emit only; it does not own the round trip. Closure happens via three paths (decision: `da005054`):
  - **Path A ‚Äî external GitHub workflows.** For a coding task: `executor.spawn(task)` opens a PR. From there, GitHub workflows handle CI + review ‚Üí automerge ‚Üí rework-cap (owned by milestone #41) ‚Üí escalate. Outcomes re-enter as fresh `events` rows via `event-dispatch.yml`. There is **no internal `pr_pipeline` module** ‚Äî the external workflows are the pipeline.
  - **Path B ‚Äî parked-event re-queue.** Blocking work parks its triggering event; a task-completion poller flips `parked` events back to `pending` once the depended-on task completes.
  - **Path C ‚Äî fresh decision event.** A cloud decision-task that resolves enters the EVENT queue as a normal event (no special internal path).
- **`SANDCASTLE_TARGET_PR`** ‚Äî single env var that switches sandcastle into rework mode. Presence ‚áí rework path (`gh pr view <N> --json headRefName` ‚Üí checkout existing branch ‚Üí apply review feedback ‚Üí push fix commits ‚Üí no new PR). Absence ‚áí fresh path (pick `status:ready` issue, new branch, new PR). Thin handle only ‚Äî sandcastle fetches review body / diff / CI state itself inside the container (stale-snapshot safety; decision: `69b7eddb-fa74-4ea3-a1bd-c03580e3023c`).

- **Readiness axis** ‚Äî scalar property of a GitHub issue: AFK-ready ‚Üî HITL-required. Orthogonal to `status:*` (workflow position) and to area/priority labels. Not stored as a single label ‚Äî *measured* by the pre-dispatch gate at the moment `/delegate` would spawn a sandcastle subagent. An issue can be `status:ready` (workflow-wise good to go) yet not AFK-ready (no AC, no decision UUID, missing `sandcastle` label). The axis is binary at dispatch time but composite at diagnosis time ‚Äî the gate reports *which* sub-check failed, so owner knows what to fix.
- **Pre-dispatch gate** ‚Äî readiness check `/delegate` runs *before* spawning any sandcastle subagent for issue #N (issue #642). Four conditions, all required: (1) `sandcastle` label present (explicit AFK-safe classification applied by `/to-issues` at slice creation per AFK-fit checklist, not auto-inferred at dispatch); (2) no `needs-*` label (`needs-grill`, `needs-research`, `needs-prd`, `needs-refactor` ‚Äî each skill removes its own label on successful completion); (3) issue body has `## Acceptance criteria` section; (4) issue body references ‚â•1 decision UUID (`[0-9a-f]{8}-[0-9a-f]{4}-‚Ä¶`). Any failure ‚áí outcome_record(`status=refused`, reason) + label `status:owner-queue` on the issue + exit; owner sees the queue at next `/status`. No Telegram escalation even on repeat refuses ‚Äî last-resort rule. Gate dominates the legacy `grill_required` checkbox in `/delegate` (decision: `6e753417`): if artefacts present, checkbox is not re-run; checkbox remains as backstop only for pre-#642 legacy issues with no artefacts and no `needs-grill` label. Lives in `/delegate` only ‚Äî interactive `/implement` is *not* gated this way (SOUL.md grill-checkbox is the in-skill backstop when operator is present). **Known fragility** (decision `6b0a5bf7`): gate fires once at `/delegate` entry, not re-validated inside the sandcastle container; pipeline changes that bypass `/delegate` would silently bypass the gate. Rationale: headless = no operator to grill against, so AFK-readiness must be verified at dispatch time, not inside the dispatched agent.
- **AFK-fit checklist** ‚Äî four-question gate `/to-issues` applies to each slice issue at creation time to decide the `sandcastle` label. Inverted-form symmetric with SOUL.md grill-checkbox: (1) declared-changed files intersect protected/safety-critical zones from per-repo `repos.conf` path-list (static grep); (2) slice requires session-loaded memory beyond what AC carries (LLM judgement on slice description); (3) mid-execution human-judgement call with no programmatic acceptance test (LLM); (4) cross-cutting / multi-repo / external-state side effects (LLM). Any yes ‚áí no `sandcastle` label ‚áí route via interactive `/implement`. Decision `9d4e0840`.
- **`status:owner-queue` label** ‚Äî pre-dispatch gate's refuse landing zone. Set on issues where `/delegate` refused AFK-dispatch (artefact missing, sandcastle label absent, blocked by `needs-*`). Surfaced by `/status` at session start so owner sees backlog of issues needing manual touch before they can re-flow through the AFK loop. Distinct from `status:ready` (workflow-ready) and `status:in-progress` (claimed). Removed by the action that fixes the cause (e.g. owner adds `sandcastle` label ‚áí also flips `status:owner-queue` ‚Üí `status:ready`).
- **`/rework` skill** ‚Äî separate from `/implement`. –ü—Ä–∏–Ω–∏–º–∞–µ—Ç PR number argument (`/rework <PR>`). Driving logic: parse structured Claude code-review verdict ‚Üí classify findings (CRITICAL/MAJOR/MINOR) ‚Üí reactive TDD per CRITICAL (failing test ‚Üí green) ‚Üí apply MAJOR fixes ‚Üí flag HIGH/CRITICAL findings —á—Ç–æ –≤–Ω–µ scope ‚Üí update PR body's `## Rework history`. ### Self-improvement / strategic-ideation (C15)

- **Strategic-ideation lane (C15)** ‚Äî the strategy-*generation* arm of self-improvement: produces NON-memory improvement proposals (new skills, prompt/instruction edits, workflows, routing, architecture, process), distinct from C5's reactive memory-mutation arm. Jarvis is an **amplifier of strategy generation, not an autonomizer of selection** ‚Äî owner + `/grill` is the permanent strategic gate; the lane never auto-selects. Two-phase milestone (decision `2ce11aaa`): **Phase-1 Consumption MVP** (visibility loop, owner-manual apply, zero autonomy) ‚Üí **Phase-2 Autonomy build-out** behind the GO-gate. Root-cause framing: prior `/self-improve` fell into disuse ‚Äî owner throttled auto-apply to M1 and the proposal menu was invisible (a *consumption* failure), with a competing recorded signal that generation produced only sharpen-the-saw increments, never cardinal moves (`self_improve_redesign` `9b453cd9`). Phase-1 deliberately isolates a falsifiable test of the consumption-first assumption ‚Äî which is why the GO-gate carries a quality dimension (below).
- **`strategic_proposal_queue`** ‚Äî sibling status-table of `memory_review_queue`. The precedent table is FK-bound to `memories(id)` (`schema.sql:912`) and so *cannot* carry non-memory proposals ‚Äî hence a sibling, not reuse. Holds the menu: status FSM (`pending/surfaced/accepted/rejected/expired`; `auto_applied` reserved for P2), candidate columns + jsonb, confidence, reasoning, reviewed_at/reviewed_by, `dismissable_until` (rejected-cooldown), undecided-expiry; rollback columns (`applied_at`/`rolled_back`/inverse-ref) reserved for P2. Drain surface = the menu disposition itself (no separate drainer).
- **Proposal candidate shape** ‚Äî `tier` (M0‚ÄìM3 risk class, **generator-assigned**, reconciled against the C15 protected map *at apply*, not at generation) + `benefit_kind ‚àà {metric | judgement}` (non-memory proposals often have no metric signal but a real judgement benefit ‚Äî life-simplification without measurable gain still counts; falsification / measurable-signal required only for tier ‚â• M2) + `traceability` field `{advances:C_x‚Üítarget | local-hygiene | exploratory/no-claim}`. **`current_trust_state` does NOT ride in the candidate** ‚Äî it is separate config that bounds what auto-applies; candidates may be any tier, the ceiling just gates application. Traceability is generator-asserted **informational only**; trust-elevation's north-star link must be **owner-confirmed** at disposition (closes the Goodhart hole where the generator supplies the input that gates its own autonomy).
- **`strategic_proposal_emitted` event** ‚Äî the lane's single operational event, on the `events` table (**not** `events_canonical`), inserted `state=processed` so `claim_next` never grabs it (the `AFTER INSERT` NOTIFY wake is a spurious LOW-cost no-op ‚Äî the orchestrator finds no `pending` row); `dedup_key` for idempotency. Consumed synchronously in-process by the generate ‚Üí dedup ‚Üí ground ‚Üí queue-insert pipeline. Review verdict is queue-table columns, not an event. All consumers deterministic ‚áí the deferred M44 orchestrator stays deferred ‚Äî this lane is permanently owner-gated and never emits a non-deterministic event.

The 4th (owner-rated value-if-applied, independent of raw acceptance count) exists so a pass distinguishes "menu now visible" from "proposals now worth accepting" ‚Äî without it the gate cannot falsify the consumption-first assumption it rests on. In P1 "applied" = owner-manual application; substance counts only owner-confirmed north-star-linked work.
- **Trust ladder (C15, Phase-2)** ‚Äî per-class, evidence-gated, **monotonic-up with auto-demote on confirmed regression** (not monotonic-only; wires `regression_suspected` ‚Üí propose-revert + tier-drop). Day-1 M0-only; after K owner-confirmed M0 cycles/class ‚Üí M1; after M successful M1 apps ‚Üí owner manually elevates to M2; M3-hard (6 fixed protected items) never auto-unlocks. **No per-class M1 elevation before the GO-gate passes AND the misimprovement-check is live.** Apply-gate inherits the C15 M3-hard/M2-strong map + the `/self-improve` never-auto-apply whitelist; any proposal touching C6/C16/C17 logic, `safety.py`, or change-budget is always principal-explicit regardless of tier or eval. Not built in P1.

### Repo-baseline (canonical GitHub-setup sync)

- **repo-baseline** ‚Äî the single canonical, installable + re-syncable GitHub-repo setup applied across all owned repos (workflows, labels, branch protection, repo settings, dependabot, community-health, issue/PR templates). Brings every repo to a uniform **format** while preserving repo-specific **content**. Replaces the prior fragmented "universal CI" (only install-time fragments existed, never re-sync; jarvis + redrobot diverged). Decisions: core mechanism `5e5f1c8f-c7cb-4e4d-bcb3-1d92894f6b18`, orchestration/rollout `0b82ccf4-491e-48d0-8f43-c75eff1d9db6`, labels/templates `261937d0-8248-4854-9dbf-471c4df71064`.
- **Managed-standalone-overwrite** ‚Äî repo-baseline's distribution mechanism. The sync writes complete workflow/config files byte-for-byte (modulo axis substitution) into each target repo ‚Äî NO `workflow_call` reusable-workflow hosting, NO composite actions, NO marked-region splicing. Chosen because for the only workflows DRY would help (code-review, merge-train) the state lives in the *consumer* (marketplace path resolves against caller's checkout; merge-train needs a GitHub-App INSTALL per repo, not a secret), `secrets: inherit` does not cross personal accounts, and redrobot is fully self-hosted (cross-account `uses:` uncallable). DRY payoff ‚âà 0 exactly where hosting is hardest.
- **3-class file model** ‚Äî every file the baseline touches is one of three classes (collapses the older AC6/AC7/AC8 split):
  - **MANAGED** ‚Äî overwritten byte-for-byte modulo axis substitution. Gate workflows, dependabot config, community-health, templates. Example: `ci-meta.yml` (#326 guard-fixture harness) is managed-uniform ‚Äî deliberately identical everywhere.
  - **LANGUAGE-TEST** ‚Äî managed *with* a `ci_language` axis (Python/Flutter/node/Java) + an escape-hatch (per-repo override file OR `test_extras` axis) for genuinely repo-specific test wiring. Example: jarvis `pytest.yml`.
  - **REPO-CUSTOM** ‚Äî listed in the manifest's `custom_files[]` allowlist; the sync never touches anything not on the allowlist (default-deny). Example: jarvis `schema-drift-check.yml`, `issue-checks.yml`. The deliberate jarvis split (managed / language / custom across separate workflow files) *is* this boundary ‚Äî it is preserved, not consolidated into one `ci.yml`.
- **Axis** ‚Äî a parameterized substitution point in a MANAGED/LANGUAGE file, value sourced from the per-repo manifest. The axis set: `required_check_contexts[]`, `runs_on[]`, `ci_language`, `code_review_marketplace`, `dependabot_ecosystems[]`, `auto_merge`, `branch_protection`, `profile` (minimal/full tier), `visibility`, `test_extras`. `required_check_contexts[]` is an explicit axis (not derived) and a meta-test pins each context name to its producing job-name (#326 class ‚Äî a rename that desyncs context‚Üîjob silently disables the gate).
- **Per-account sync pass** ‚Äî orchestration runs once per GitHub account with that account's credential (Osasuwu pass over its 5 repos, SergazyNarynov pass over redrobot). A single flat loop is impossible: the active `gh` token is Osasuwu-only without `read:org`, so it 404/403s on SergazyNarynov. redrobot's pass carries `auto_merge=off` + `branch_protection=off` axes (free private plan, memory `170f8018`).
- **Ordering invariant** ‚Äî within a pass: distribute workflows FIRST, apply branch-protection contexts SECOND. Protection that references a not-yet-existing check deadlocks a bare repo (dnd-calendar / farming-evolution have no workflows yet ‚Üí protection on never-run checks blocks forever).
- **Audit-first slice** ‚Äî repo-baseline's first implementation slice is an empirical 6-repo dump (real labels / workflows / settings / protection) that POPULATES manifest axis values from reality. The premortem's earliest-prevention point: modelling variation with fixed axes BEFORE observing live divergence would produce a sync that only works on jarvis (the repo whose shape seeded the model).
- **Destructive-once label migration** ‚Äî the clean label schema is reached by a one-time migration (pure rename-in-place via `gh label edit --name`, which preserves issue/PR associations; collision-on-target ‚Üí merge + delete-orphan with owner confirm; many‚Üíone consolidation), AFTER which routine sync is additive-only (never deletes ad-hoc labels). Separates the AC14 "clean schema" goal from the AC5 "non-destructive routine" goal so they stop conflicting.
- **community-health source boundary** ‚Äî non-overlapping single-source rule. For the 5 Osasuwu repos, community-health + default templates come ONLY from the `Osasuwu/.github` special repo (zero-cost GitHub fallback when the repo has no local copy) ‚Äî so existing local copies are removed in migration and the sync does NOT write them there (two sources would conflict). For redrobot (`.github` cannot reach cross-account), the sync writes community-health locally. `.github` covers ONLY community-health + templates ‚Äî labels / workflows / settings / protection / dependabot are sync-owned everywhere; the hope that `.github` removes "half the common layer" is false, it removes only the community-health subset.

### Workflow vocabulary

- **Acceptance criteria (AC)** ‚Äî buffer between scope and tests. Must be **literally verifiable**: not "handles edge cases" but "given input X, output Y; given empty input, raises ValueError E". Source of test cases.
- **Vertical slice / tracer bullet** ‚Äî task that crosses the entire stack (schema ‚Üí service ‚Üí API ‚Üí UI ‚Üí tests) to a verifiable end-state. Default decomposition unit per `/to-issues` skill.
- **Smart zone** ‚Äî context window region where reasoning quality is high (~first 100K tokens). Past it = "dumb zone". Triggers Plan/Execute/Clear ritual.
- **Plan / Execute / Clear** ‚Äî long-session rhythm: write plan, execute against it, dump state to memory, start fresh window for next phase.
- **Deep module** ‚Äî small interface, large hidden implementation. Caller knows minimum, gets maximum behavior. Anti-pattern: shallow modules where interface ‚âà implementation complexity.
- **Deletion test** ‚Äî diagnostic for module depth: imagine deleting it. If complexity vanishes, it was a pass-through (shallow). If complexity reappears across N callers, it earned its keep (deep).
- **Implicit assumption** ‚Äî domain rule that's "obvious" to the human but not in writing. Source of scope shrinkage. Surfaced via `/grill`, fixed by adding to this file or to AC.
- **Sycophancy** ‚Äî model's tendency to agree with user's proposal regardless of correctness. Industry baseline ~63.7% under user-opinion exposure ([arxiv 2508.02087](https://arxiv.org/html/2508.02087v1)). Not a tone issue, a correctness issue: drives scope shrinkage and missed alternatives. Mitigations stack ‚Äî see milestone #43 and decision `316c5911-9f06-44de-8f99-20fe3e9fa448`.
- **Personalization-sycophancy paradox** ‚Äî heavy user-modeling (SOUL.md, always_load memory, calibration to owner tendencies) *increases* agreement bias ([MIT 2026](https://news.mit.edu/2026/personalization-features-can-make-llms-more-agreeable-0218), [ICLR 2026](https://openreview.net/pdf?id=igbRHKEiAs)). Implication: identity layer must be deliberately suspended on consequential decisions, not just on stylistic ones.
- **Cross-context review** ‚Äî anti-sycophancy mechanism: at critical forks, dispatch a subagent with scrubbed context (no SOUL/memory/CONTEXT) to critique a proposal cold. Single-agent self-critique is grading-own-exam; role-isolation breaks the personalization feedback loop.
- **4-channel research intake** ‚Äî mandatory protocol for `/research` and pre-grill: parallel pulls across (1) end-user experience, (2) domain specialists, (3) quantitative data, (4) adversarial/failure-mode reports. Memory recall does NOT substitute ‚Äî recall is past convergence, not fresh signal. Decision `6fd2df1d-defc-440d-ba30-71880409e533`.
- **Research-pass gate** ‚Äî shared precondition fragment (`_shared/research-pass-gate.md`) injected before the finalization phase of `/grill` (Phase 3 entry), `/reason` (resolution phase), `/to-prd` (PRD publication), and `/improve-codebase-architecture` (child-issue creation). Fires only for high-stakes decisions: `reversibility ‚àà {hard, irreversible}` OR `confidence < 0.7`. Blocks proceed unless a 4-channel research artifact (`source_provenance LIKE 'skill:research%'`, ‚â§60d) exists for the topic. Override: explicit owner waiver recorded as event/memory so `/learn` can spot over-use. Issue #693 / PR#728. Decision `7acf0156-a259-40dc-beed-75d66b57bbf0`.
- **TDD-mode** ‚Äî operating mode of `/implement` and `/delegate` that runs the red‚Üígreen‚Üírefactor loop one acceptance-criterion at a time. Engaged after the SOUL.md grill-me checkbox fires and a `/grill` has resolved the AC. Reference material in `.claude-userlevel/skills/_shared/tdd/`.
- **Testable interface** ‚Äî interface designed so behavior can be verified without reaching into implementation. Three rules: (1) accept dependencies as parameters, don't construct them inside; (2) return results rather than producing hidden side effects; (3) keep surface area small (fewer methods + fewer params = simpler test setup). Operational counterpart to "deep module" ‚Äî a deep module with a hard-to-test interface is still a defect.

### Skill trigger model (ADR-0001)

- **Type 1 trigger** ‚Äî event/cron-driven skill invocation (Stop hook, SessionStart, scheduled cron, GitHub webhook). The skill fires in a fresh session, deterministically, without the model deciding. Examples: `/cycle`, `/learn`, `/end`.
- **Type 2 trigger** ‚Äî user or orchestrator supplies an intent-shaped prompt at session start; the model matches the skill description and invokes. Both human-typed and headless-orchestrator-issued prompts are Type 2. Examples: `/grill`, `/implement`, `/diagnose`.
- **Type 3 trigger** ‚Äî mid-task self-trigger by the model. **Not designed for.** Skill invocation mid-task eats smart-zone budget for the current task and empirically fires unreliably. Let the current task finish; the orchestrator triggers the next skill in a fresh session.

## Invariants (domain rules that must always hold)

These are the "obvious" assumptions that previously bit because they weren't written down. Add to this list every time a `/grill` session surfaces one.

- **Threat-model duality** ‚Äî defence layers must match the threat model, not stack defensively for "more is better". Sandcastle is already process-isolated by Docker + sterile image; piling host-grade defences on top adds friction without adding security. Cross-link memory `enforcement_layer_matches_threat_model`.

### Memory & persistence

- **Memory is cross-device source of truth.** Anything important goes through Supabase. File-based memory (`~/.claude/projects/.../memory/`) is device-local and does NOT sync.
- **Every `memory_store` carries `source_provenance`.** No exceptions. Server rejects unattributed writes (JTMS attribution requirement).
- **`memory_store` is idempotent on `(project, name)` ‚Äî no similarity threshold ever blocks a write.** Project-scoped writes go through atomic upsert on the unique constraint; global-scoped (project=null) writes do an explicit SELECT-then-UPDATE/INSERT in the same handler. Auto-link, consolidation, and classifier UPDATE-supersession all run *after* the write has landed and are best-effort ‚Äî they cannot reject a candidate. Consolidation candidates are advisory and surface in the structured response (`stored`, `action`, `memory_id`, `project`, `consolidation_candidates`, `classifier_pending`, `message`) so callers never infer success from prose. Filed #658 after an agent confabulated a "dup-detector block" against a prose-only success message.
- **Sandcastle provenance gate is table-level + op-level, not agent-level.** RLS on `memories` / `task_outcomes` / `episodes` / `events_canonical` requires `source_provenance` (or `actor`, on episodes/events) `LIKE 'sandcastle:%'` for **every** anon INSERT/UPDATE/DELETE ‚Äî not just INSERT. Slice 3 (#542) gated INSERT; slice 3.5 (#565) extended to UPDATE+DELETE so anon can neither wipe rows nor forge/erase the provenance column. Service-role bypasses RLS ‚Äî host MCP must use `SUPABASE_SERVICE_KEY` (#564, #569) to write any non-sandcastle provenance.
- **State is never in static files or memory.** Status %, dates, PR markers, "current sprint" ‚Äî all of these live in GitHub. Static storage is for stable knowledge, not state.
- **`record_decision` always passes `memories_used=[<UUIDs>]`.** Names, not UUIDs, break attribution.
- **Memory hygiene is owner-invoked, never autonomous.** No PreToolUse / UserPromptSubmit / Stop hook fires "mark stale?" hints or auto-demotes rows. `/curate` (M45 #768) runs only on explicit command. Decision `d5bfd444-78f3-4fca-bcd8-f392f647504c` supersedes `no_memory_hygiene_tool` (`719fb533`, 2026-04-15) ‚Äî preserves the 2026-04-15 banner-blindness rationale by routing all destructive lifecycle writes (`expired_at`, `superseded_by`) through owner-confirmed Curation, not automated detection.

### Skills & infra

- **Skills are universal**, not project-specific. They live in `.claude-userlevel/skills/`. Project-specific skills go in `<project>/.claude/skills/` (rare ‚Äî currently only `/sprint-report` for redrobot).
- **`.claude-userlevel/` is canonical**, `~/.claude/` is mirror. Edits to mirror drift from source on next install.
- **`config/SOUL.md` is identity for THIS jarvis instance.** Currently single Jarvis = single SOUL.

The reactive-core `orchestrator` runs **routing-policy only** ‚Äî a thin instruction set deciding among the three dispositions ‚Äî it does **not** load the full SOUL (it is not the principal, it is the router). Per-agent SOUL split (`config/agents/<name>/SOUL.md`) is **deferred**: it depends on federation, and federation itself is deferred until reactive-native core ships and proves insufficient (decision: `9757b985`).
