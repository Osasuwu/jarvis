## Findings

### FINDING-001: Installer tests use string-matching oracles for command verification
**Severity:** MEDIUM
**File:** `tests/test_installer.py`
**Detail:** Tests verify that the correct shell commands would be generated by asserting on string representations (e.g., `assert "command_name" in result`). These oracles are brittle — a formatting change (spacing, quoting, flag order) would break them. A structured oracle (parsed command AST or structured result object) would be more robust.

### FINDING-002: Portability oracles are regex-based and well-constructed
**Severity:** (POSITIVE)
**File:** `tests/install/test_install_script_portable.py`
**Detail:** The regex patterns for detecting hardcoded paths are specific and well-annotated. The Join-Path multi-arg guard, NSSM autodiscovery, and SeServiceLogonRight checks are clear oracles with minimal false-positive risk.

### FINDING-003: No property-based or fuzz testing for installer edge cases
**Severity:** LOW
**File:** `tests/test_installer.py`
**Detail:** All test inputs are hand-crafted. Given the combinatorial nature of installer paths (OS combinations, install modes, skip flags), property-based testing (e.g., Hypothesis) would provide stronger coverage guarantees than the current example-based approach.
